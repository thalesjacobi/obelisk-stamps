Facebook Posting

Project: Obelisk Stamps File: Technical Reference Date: March 2026

1. Overview

The Facebook posting module allows publishing content from the admin article editor directly to a Facebook Page. It supports three content types:

  • Carousel — Multi-photo post using generated carousel images (up to 10 photos)
  • Cinemagraph — Single video post using a cinemagraph clip
  • Narrated Video — Single video post using a generated narrated MP4

Posts are published via the Facebook Graph API v21.0 in background threads. The frontend polls for status updates and displays a progress bar until completion. Successful posts can be archived or deleted from the admin UI.

2. Prerequisites

Environment Variables

VariableDescriptionRequired
FB_PAGE_ID The numeric ID of the Facebook Page to post to (e.g. 911333915406357) Yes
FB_PAGE_ACCESS_TOKEN A permanent Page Access Token with pages_manage_posts permission Yes

Both variables must be set for the Facebook sections to appear in the article editor. They are stored in GitHub Secrets and passed to Cloud Run at deploy time via the GitHub Actions workflow (.github/workflows/main.yml).

Meta App Setup

  1. Create a Meta App at developers.facebook.com
  2. Add the Facebook Login for Business use case
  3. Request these permissions: pages_show_list, pages_read_engagement, pages_manage_posts
  4. Publish the app (Settings → Publish). In Development mode, tokens expire after a few hours even if generated as "long-lived"
Important: The Meta app must be in Live (Published) mode. Development mode tokens expire rapidly and will cause posting failures.

3. Token Management

Token Types

TypeLifespanUse
Short-lived User Token ~1–2 hours Generated by Graph API Explorer; used only to obtain a long-lived token
Long-lived User Token ~60 days Intermediate step; used only to obtain a permanent Page Token
Page Access Token Never expires (when derived from a long-lived user token on a Published app) This is what the app uses as FB_PAGE_ACCESS_TOKEN

Generating a Permanent Page Token

Follow these three steps in the Graph API Explorer:

Step 1: Get a fresh User Token

  1. Open the Graph API Explorer
  2. Select your app (Obelisk Stamps) in the top-right dropdown
  3. Ensure permissions include: pages_show_list, pages_read_engagement, pages_manage_posts
  4. Click "Generate Access Token" (green button)
  5. Authorise when prompted — you now have a short-lived User Token in the Access Token field

Step 2: Exchange for a Long-lived User Token

  1. In the URL bar, enter: 
    /oauth/access_token?grant_type=fb_exchange_token
  &client_id=YOUR_APP_ID
  &client_secret=YOUR_APP_SECRET
  &fb_exchange_token=CURRENT_USER_TOKEN
  2. Replace YOUR_APP_ID with your App ID (found in App Settings → Basic)
  3. Replace YOUR_APP_SECRET with your App Secret (found in App Settings → Basic, click "Show")
  4. Replace CURRENT_USER_TOKEN with the token currently in the Access Token field
  5. Click Submit
  6. The response will contain an access_token with expires_in: ~5184000 (~60 days)

Step 3: Get the Permanent Page Token

  1. Copy the long-lived user token from the Step 2 response
  2. Paste it into the Access Token field on the right side panel (replacing the old token)
  3. In the URL bar, enter: 
    /YOUR_PAGE_ID?fields=access_token

    (e.g. /911333915406357?fields=access_token)

  4. Click Submit
  5. The response contains the permanent Page Access Token
Note: The ?fields=access_token in the URL is the literal text — it tells the API "return the access_token field for this Page". It is not where you paste your token. Your token goes in the Access Token field on the right panel.

Deploying the Token

  1. Copy the permanent Page Token from Step 3
  2. Update it in GitHub Secrets: Repository → Settings → Secrets and variables → Actions → FB_PAGE_ACCESS_TOKEN
  3. Trigger a deploy by pushing a commit (the GitHub Actions workflow passes secrets to Cloud Run via --set-env-vars)
  4. Verify: after deploy, open an article editor and check that the Facebook section loads without errors

Debugging Tokens

Use the Access Token Debugger to inspect any token. Paste the token and check:

  • Type: should say "Page"
  • Expires: should say "Never" for a permanent page token
  • Valid: should be "True"
  • Scopes: should include pages_manage_posts

Troubleshooting

SymptomCauseFix
"Session has expired" error Token was generated while app was in Development mode, or a User Token was used instead of a Page Token Publish the app, then regenerate the token following all three steps above
Token expires after ~2 hours Short-lived User Token used directly as FB_PAGE_ACCESS_TOKEN Complete all three steps — you need the Page Token from Step 3, not the User Token
Token expires after ~60 days Long-lived User Token used instead of Page Token Complete Step 3 to convert to a permanent Page Token
"Error validating access token" after deploy Cloud Run still running old container with cached env vars Push a new commit to trigger a fresh deploy, or manually deploy a new revision in Cloud Run console
Facebook sections not visible in editor FB_PAGE_ID or FB_PAGE_ACCESS_TOKEN not set in environment Add both to GitHub Secrets and redeploy

  "caption": "The posted caption text...",
  "post_type": "car",          // or "cine"
  "posted_at": "2026-03-15T14:25:00Z",

}

5. Backend API Routes

All routes require @login_required and @admin_required.

Post to Facebook

POST /admin/articles/<id>/post-to-facebook

Spawns a background worker to upload and publish content.

ParameterTypeDescription
captionstringPost caption (max 2200 chars)
typestring"car" (carousel) or "cine" (cinemagraph)

Response: {"ok": true} or {"error": "..."}

Check Post Status

GET /admin/articles/<id>/fb-post-status?type={car|cine}

Returns current posting status. Polled by frontend every 5 seconds.

// Response
{
  "status": "running:upload:3/10",
  "result": "",
  "media_id": "",
  "history": [...],
  "caption": ""      // populated from snapshot when post is done
}

Check if Post is Live

GET /admin/articles/<id>/check-facebook-post?type={car|cine}

Queries the Facebook Graph API to verify the post still exists.

// Response
{
  "live": true,
  "permalink": "https://www.facebook.com/...",
  "reason": ""       // populated when live=false
}

Edit Caption

POST /admin/articles/<id>/edit-facebook-caption

Updates the caption of a published post via the Graph API.

ParameterTypeDescription
captionstringNew caption (max 2200 chars)
typestring"car" or "cine"

Archive Post

POST /admin/articles/<id>/archive-facebook-post

Archives the post locally (moves to history). Does not delete from Facebook.

ParameterTypeDescription
typestring"car" or "cine"

Delete and Archive Post

POST /admin/articles/<id>/delete-facebook-post

Deletes the post from Facebook via the Graph API, then archives locally.

ParameterTypeDescription
typestring"car" or "cine"

6. Background Workers

Carousel / Cinemagraph Worker

_post_to_facebook_worker(article_id, caption, post_type="cine")

Runs in a background thread spawned by the post route.

Carousel Flow (post_type="car")

  1. Set status to running:0
  2. Fetch article data (title, carousel_images, carousel_punchlines)
  3. For each image (up to 10): upload to /{page_id}/photos with published=false
  4. Update status to running:upload:N/M after each upload
  5. Set status to running:publishing
  6. Create feed post via /{page_id}/feed with all photo IDs as attached_media
  7. Save post ID, permalink, and snapshot
  8. Set result to done:{permalink}

Cinemagraph Flow (post_type="cine")

  1. Set status to running:0
  2. Fetch article data
  3. Set status to running:upload
  4. Upload video to /{page_id}/videos with file_url and description
  5. Save post ID and permalink
  6. Set result to done:{permalink}

Narrated Video Worker

_post_narrated_fb_worker(article_id, video_url, caption, run_ts)

  1. Set status to running:upload
  2. Upload video to /{page_id}/videos with file_url and description
  3. Save media ID
  4. Set result to done:{permalink}
Error handling: All workers set status to idle and result to error:{message} on failure. Full tracebacks are logged to the activity log.

8. Logic Flows

A. Posting Flow

  1. User enters caption in textarea (max 2200 characters)
  2. User clicks "Post to Facebook" button
  3. Frontend disables button, shows progress bar
  4. POST to backend → backend spawns background worker thread
  5. Frontend polls /fb-post-status every 5 seconds
  6. Progress bar updates based on status string
  7. On done:{permalink}: show success UI, populate caption as read-only, disable "Copy from Instagram" link
  8. On error:{message}: show error alert, re-enable post button

B. Archive Flow

  1. User clicks "Archive"
  2. POST to /archive-facebook-post
  3. Backend moves snapshot to history array in site_settings, clears status/result/media_id
  4. Frontend resets UI to initial state, renders updated history

C. Delete and Archive Flow

  1. User clicks "Delete and archive post"
  2. Browser confirmation dialog shown
  3. POST to /delete-facebook-post
  4. Backend sends DELETE to Facebook Graph API (/{media_id})
  5. Backend moves snapshot to history, clears current state
  6. Frontend resets UI, renders updated history

D. Page Load Restore

  1. checkFbRunning(mode) called on page load
  2. Fetches /fb-post-status
  3. If result is done:{permalink}: call showFbDone()
  4. If status is running:* and no error result: resume polling
  5. If status is running:* but result is error:*: skip (stale state, don't show error on load)
  6. Render history if present
Stale error handling: If the worker crashed and left status as "running" but the result is an error, the frontend does not show the error on page load. Errors are only displayed when the user actively triggers a posting action.

9. Error Handling

Backend Workers

  • All error paths set status to idle and result to error:{message}
  • Full exception tracebacks are logged to the activity log
  • HTTP timeouts: 60s for photo uploads, 120–180s for video uploads, 15s for status checks

API Routes

  • Return 400 if FB credentials not configured
  • Return 400 for validation errors (empty caption, missing parameters)
  • Return 500 on unexpected exceptions

Frontend

  • Polling .catch() handlers retry silently (don't fail the entire flow)
  • Error messages displayed in Bootstrap alert-danger divs
  • Post button re-enabled on error so the user can retry
  • Stale errors from previous sessions are not shown on page load

11. Facebook Graph API Reference

Base URL: https://graph.facebook.com/v21.0

Endpoints Used

ActionMethodEndpointKey Parameters
Upload photo (unpublished) POST /{page_id}/photos url, published=false, access_token
Create carousel post POST /{page_id}/feed message, attached_media[N], access_token
Upload video POST /{page_id}/videos file_url, description, access_token
Update caption POST /{media_id} message, access_token
Check post exists GET /{media_id}?fields=id access_token
Delete post POST /{media_id} (DELETE method) access_token

Required Permissions

  • pages_show_list — List pages the user manages
  • pages_read_engagement — Read page content and engagement
  • pages_manage_posts — Create, edit, and delete posts on the page

Useful Tools

  • Graph API Explorer — Test API calls and generate tokens
  • Access Token Debugger — Inspect token type, expiry, and permissions
