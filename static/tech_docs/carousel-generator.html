Instagram Carousel Generator

Project: Obelisk Stamps File: Technical Reference Date: March 2026

1. Overview

The Instagram Carousel Generator is an admin tool that transforms an article's content into a set of up to 10 square (1024×1024) images suitable for an Instagram carousel post. It uses a two-step AI pipeline:

  1. GPT-4.1-mini reads the article and writes N sequential scene descriptions — one DALL-E prompt and one short punchline (≤15 words) per scene.
  2. DALL-E 3 generates one image per prompt, with a consistent visual style suffix appended to every prompt.

Results stream to the browser in real time via Server-Sent Events (SSE).

Once images are generated, the tool supports posting the carousel directly to Instagram via the Instagram Graph API. The posting flow composes each slide image (adding the punchline as a text overlay), uploads the composed JPEGs to GCS, creates Instagram child containers, polls until each container is ready, creates a carousel container, and publishes it. A separate GPT-4.1-mini call can generate an optimised Instagram caption before posting.

All significant actions (caption generation, posting attempts, errors, archive operations, live-checks) are written to a per-article activity log stored in site_settings.

3. Database Schema

Table: articles — carousel columns

Four TEXT columns are added to the existing articles table to store carousel data.

ColumnTypePurpose
carousel_promptsTEXT (JSON array)DALL-E 3 prompts. Indices 0–9 = current; 10+ = archived.
carousel_imagesTEXT (JSON array)GCS public URLs (or relative file paths when GCS is not configured). Same index convention.
carousel_punchlinesTEXT (JSON array)Short Instagram captions (≤15 words). Same index convention.
carousel_styleTEXTThe style suffix that was active when this article's carousel was last generated. Used by "View Style" modal.
Index convention: Slots 0–9 are the active/current carousel. Slots 10+ are archived (moved there by "Archive All" or per-image re-run). A null / None value in slots 0–9 means that slot is empty and eligible for generation.

Migration

The carousel columns are added at startup via an idempotent migration in init_articles_table(). Each ALTER TABLE ... ADD COLUMN is wrapped in a try/except so it silently skips if the column already exists: 

for col in ("carousel_prompts TEXT", "carousel_images TEXT",
            "carousel_punchlines TEXT", "carousel_style TEXT"):
    try:
        cur.execute(f"ALTER TABLE articles ADD COLUMN {col}")
    except Exception:
        pass  # column already exists

All Instagram posting state is stored transiently in site_settings. The helper function _ig_keys(post_type, article_id) returns a dict of key names for a given article and post type ("car" for carousel, "cine" for cinemagraph).

Helper Functions

get_setting(key, default=None)

Reads a single value from site_settings. Returns default if the key doesn't exist.

def get_setting(key, default=None):
    """Return a value from site_settings, or default if not found."""
    row = query_one("SELECT value FROM site_settings WHERE `key` = %s", (key,))
    return row[0] if row else default

upload_bytes_to_gcs(data, object_name, content_type=None)

Uploads raw bytes to GCS and returns the public HTTPS URL. Returns None if _gcs_bucket is not initialised (i.e., GCS_BUCKET_NAME is not set). The object name is prefixed with GCS_PATH_PREFIX if configured.

_ig_keys(post_type, article_id)

Returns a dict of site_settings key names for the given post_type ("car" or "cine") and article ID. 

def _ig_keys(post_type, article_id):
    prefix = "ig_car_post_" if post_type == "car" else "ig_post_"
    return {
        "status":   f"{prefix}status_{article_id}",
        "result":   f"{prefix}result_{article_id}",
        "media_id": f"{prefix}media_id_{article_id}",
        "history":  f"{prefix}history_{article_id}",
        "snapshot": f"{prefix}snapshot_{article_id}",
    }

_ig_api_call(method, url, **kwargs)

Wrapper around requests that performs one automatic retry on rate-limit. If the response JSON contains an error message with "request limit", the function waits 30 seconds and retries once before returning.

_add_activity_log(article_id, title, content, component=None)

Appends a log entry to the article's activity log in site_settings. See Section 7 — Activity Log for full details.

Init Functions

init_articles_table()

Creates the articles table if it doesn't exist, then runs the idempotent carousel column migration shown in the Database section above. Called once at Flask startup.

init_site_settings()

Creates the site_settings table if it doesn't exist, then seeds the default carousel_style value on first run. INSERT IGNORE ensures the default is only written once — subsequent starts leave any admin-edited value untouched. 

def init_site_settings():
    conn = get_db()
    cur = conn.cursor()
    cur.execute("""
        CREATE TABLE IF NOT EXISTS site_settings (
            `key` VARCHAR(100) PRIMARY KEY,
            value TEXT,
            updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
        )
    """)
    cur.execute(
        "INSERT IGNORE INTO site_settings (`key`, value) VALUES ('carousel_style', %s)",
        (CAROUSEL_STYLE_SUFFIX,),
    )
    conn.commit()
    cur.close()
    conn.close()

API Routes — Image Generation

All routes require @login_required and @admin_required decorators.

MethodURLFunctionReturns
GET /admin/articles/<id>/carousel-images admin_article_carousel_images JSON: {images, prompts, punchlines}
POST /admin/articles/<id>/generate-carousel admin_article_generate_carousel SSE stream (text/event-stream)
POST /admin/articles/<id>/archive-carousel admin_article_archive_carousel JSON: {success, archived}
POST /admin/articles/<id>/regenerate-carousel-image admin_article_regenerate_carousel_image JSON: {url, prompt, punchline}
GET /admin/settings admin_settings Renders admin.html with active_tab="settings"
POST /admin/settings/save admin_settings_save JSON: {success: true} or {error: "..."}

Route detail: GET carousel-images

Reads carousel_prompts, carousel_images, carousel_punchlines from the articles row. For local paths, converts relative image paths to full static URLs via url_for("static", filename=...). GCS URLs are returned as-is (they are already absolute public URLs). Returns arrays for all three; null entries in the arrays indicate empty slots.

Route detail: POST generate-carousel

The main generation endpoint. Accepts no request body. Returns a streaming SSE response (see SSE Event Protocol below).

Key pre-flight logic (outside the streaming closure):

  • Reads active_style from site_settings.carousel_style (falls back to constant).
  • Loads existing carousel_images array, pads to 10 with None.
  • Computes empty_slots = list of 0-based indices where the image is None/null.
  • N = len(empty_slots). If N == 0, yields done immediately and returns.

After each DALL-E call, the image bytes are saved to the local filesystem AND uploaded to GCS via upload_bytes_to_gcs(). If GCS is configured, the returned public URL replaces the local relative path in the DB; if not, the local relative path is stored.

Route detail: POST archive-carousel

Moves non-null items from slots 0–9 to the end of the arrays (after any existing archived items). Slots 0–9 are set to None. 

# After archiving, the arrays look like:
# [None, None, ..., None,   <- 10 empty current slots
#  <previously archived items>,
#  <newly archived items from slots 0-9>]

Returns {success: true, archived: N} where N is the count of items moved.

Route detail: POST regenerate-carousel-image

Request body: {"index": 0–9, "prompt": "...", "punchline": "..."}

  • Only indices 0–9 are valid (active slots).
  • New image saved as image_{index+1}_{timestamp}.png to preserve the old file.
  • Image bytes uploaded to GCS. GCS URL stored in DB if upload succeeds; local relative path used as fallback.
  • Old prompt/image/punchline at that index are appended to the end of the arrays (archived).
  • The slot is updated in-place with the new values.
  • Also saves carousel_style = active_style to the article row.
  • Writes to the activity log on both success and failure.

Route detail: POST /admin/settings/save

Accepts {"carousel_style": "..."}. Uses MySQL upsert pattern: INSERT INTO ... ON DUPLICATE KEY UPDATE value = %s. Validates that the value is non-empty before writing.

Route detail: POST post-to-instagram

Pre-flight checks before spawning the background thread:

  • Returns 400 if IG_USER_ID or IG_ACCESS_TOKEN env vars are not set.
  • Returns 409 {error: "already_running"} if ig_car_post_status_{id} already starts with "running".
  • For post_type = "car": loads carousel_images[:10] and returns 400 if fewer than 2 non-null images exist.
  • Clears the stale result key, seeds status to "running:starting", then launches _post_to_instagram_worker in a daemon thread.
  • Caption is trimmed to 2200 characters (Instagram limit).

Route detail: GET ig-post-status

Returns the current values of all four keys for the given post type: status, result, media_id, and history (JSON-decoded). The frontend polls this endpoint every 5 seconds while a post is in progress, and also on page load to restore any previously posted state.

Route detail: GET check-instagram-post

Reads ig_car_post_media_id_{id} and calls the Instagram Graph API (GET /{media_id}?fields=id,permalink). If the API returns an id field, the post is considered live. A back-fill path handles older posts that stored the permalink in the result key before media_id was saved separately — the numeric ID is extracted from the permalink URL and persisted to the media_id key for future calls.

Route detail: POST archive-ig-post

Reads the current media_id, result, and snapshot keys. Appends a new entry to the history array. Clears media_id, result, and snapshot keys (set to empty string). Returns the full updated history array.

Route detail: POST edit-instagram-caption

Updates the caption of the most recently published post. Uses POST /{media_id} with {caption, access_token}. Requires media_id to be stored — will attempt the same back-fill from the result URL as the check-live route.

Background Worker: _post_to_instagram_worker

Runs in a daemon thread. Handles both post_type="car" (static carousel images) and post_type="cine" (cinemagraph video clips). This section focuses on the car path; see the Cinemagraph technical document for the cine path.

The worker uses three internal helpers that write directly to site_settings: _set_status(s), _set_result(s), and _clear_status(). Status is always cleared at the end of every code path (success, failure, or exception); result persists until the next run or archive.

5. SSE Event Protocol

The generate-carousel endpoint returns Content-Type: text/event-stream. Each event has the format: 

event: <type>
data: <JSON string>
Event typePayload fieldsWhen emitted
prompts {prompts: string[], punchlines: string[]} Immediately after GPT returns all N scene descriptions. Allows the UI to show "Generating images…" state.
image {index, seq, of, url, prompt, punchline} After each DALL-E image is downloaded, saved to disk, and uploaded to GCS. index = 1-based slot number; seq = position in current generation sequence; of = total images being generated (N). url is the GCS public URL if GCS is configured, otherwise a static file URL.
error {index, message} When a single image fails (content policy, rate limit, etc.). Generation continues for remaining slots.
done {count} After all slots are processed. count = number of successfully generated images.
Why not EventSource? The browser's built-in EventSource API only supports GET requests. Since this endpoint uses POST, the frontend uses fetch() + ReadableStream to consume the SSE stream manually. See JS: SSE Streaming Pattern.

Progress calculation (frontend)

// In handleCarouselEvent(), on 'image' event:
const pct = p.of ? Math.round((p.seq / p.of) * 100) : (p.index * 10);
progressBar.style.width = pct + '%';
statusText.textContent = 'Generated ' + (p.seq || p.index) + ' of ' + (p.of || 10) + '…';

6. Instagram Posting

The Instagram posting panel in the article edit UI allows an admin to compose a caption and post the carousel images as an Instagram carousel post. The entire posting operation runs in a background thread. The frontend polls for status every 5 seconds.

Caption Generation

The POST /admin/articles/<id>/generate-ig-caption endpoint generates an Instagram caption using GPT-4.1-mini. It is shared between the Carousel and Cinemagraph components — the optional component field in the request body controls which component name is recorded in the activity log.

The caption generation process:

  1. Loads title, content, and slug from the article row.
  2. Strips HTML tags from content and truncates plain text to 500 characters.
  3. Reads the system prompt from site_settings: first checks ig_caption_prompt_{article_id} (per-article override), then ig_caption_prompt (global override), then falls back to the hardcoded _IG_CAPTION_DEFAULT_PROMPT constant.
  4. Calls GPT-4.1-mini with max_tokens=600.
  5. If SITE_URL and slug are available, appends a call-to-action line: "Want to read the full article? Access the following link:\n{url}".
  6. Logs the action and returns {caption: "..."}.
Caption prompt customisation: The system prompt can be overridden at two levels. A global override in site_settings under the key ig_caption_prompt applies to all articles. A per-article override under ig_caption_prompt_{article_id} takes the highest priority. If neither is set, the built-in default prompt is used.

Polling Pattern

Because the Instagram posting worker runs in a background thread (and may take several minutes), the frontend uses a polling loop rather than SSE. 

// Frontend JS — polling loop (simplified)
let igPollTimer = null;

function startIgPolling() {
    igPollTimer = setInterval(async () => {
        const r = await fetch(`/admin/articles/${ARTICLE_ID}/ig-post-status?type=car`);
        const d = await r.json();
        handleIgStatus(d.status, d.result, d.media_id);
        if (!d.status || !d.status.startsWith('running')) {
            clearInterval(igPollTimer);
            igPollTimer = null;
        }
    }, 5000);
}

function handleIgStatus(status, result, media_id) {
    if (!status || !status.startsWith('running')) {
        // Update UI based on result
        if (result && result.startsWith('done:')) {
            showIgSuccess(result.slice(5), media_id);  // permalink
        } else if (result && result.startsWith('error:')) {
            showIgError(result.slice(6));
        } else if (result && result.startsWith('warn:')) {
            showIgWarning(result.slice(5));
        }
        return;
    }
    // Show progress label from status string
    showIgProgress(status);
}

Status Key Values

The ig_car_post_status_{id} key is set by the worker to reflect the current phase of the posting operation. The frontend converts these to human-readable progress labels.

Status valuePhase
running:startingThread spawned; not yet reached the worker body
running:compose:{i}/{n}Composing slide i of n (downloading image, applying punchline overlay, uploading to GCS) — image-carousel path only
running:containers:{i}/{n}Creating Instagram child container i of n via the Graph API
running:poll:{i}/{n}Polling child container i of n until its status_code is FINISHED
running:carouselCreating the top-level carousel container
running:carousel_pollPolling the carousel container until FINISHED
running:publishCalling media_publish on the carousel container ID
running:publish_verifyPublish returned an error; checking recent media to see if the post went through anyway

Status key lifecycle: The status key is deleted (DELETE FROM site_settings WHERE key = ...) at the end of every worker run — whether successful, failed, or an unhandled exception. This ensures that the absence of the status key (or a non-running value) is always a reliable indicator that the worker has finished.

Result Key Values

Result prefixMeaningFrontend action

error:{message} Posting failed. The value after the colon is a human-readable error message. Show error alert with the message
warn:{message} Rate-limit ambiguity: the publish API call returned a rate-limit error, but the post may have gone through. The carousel container ID was saved. Show warning alert; show "Check if live" and "Archive post" buttons

Post Snapshot

After a successful publish (and also in the rate-limit warn case), the worker saves a snapshot of what was posted to ig_car_post_snapshot_{id}. The snapshot is a JSON object: 

{
    "caption":    "The full Instagram caption text",
    "image_urls": ["https://storage.googleapis.com/…/image_1.png", …],  // static carousel path
    "video_urls": [],             // empty for car post_type
    "post_type":  "car",
    "posted_at":  "2026-03-14T10:30:00Z"   // UTC ISO 8601
}

The snapshot is read by the archive route to populate the history entry. It is cleared (set to "") when the post is archived.

Check if Live

The GET /admin/articles/<id>/check-instagram-post?type=car route queries the Instagram Graph API to verify the post still exists: 

GET https://graph.instagram.com/v22.0/{media_id}
    ?fields=id,permalink
    &access_token={IG_ACCESS_TOKEN}

If the response contains an id field, the route returns {live: true, permalink: "https://www.instagram.com/p/..."}. Otherwise it returns {live: false, api_error: "..."} with the cleaned-up Graph API error message (the boilerplate "Please read the Graph API documentation at …" suffix is stripped).

Both outcomes are written to the activity log with title "Check Post Live (car)" and component "carousel".

Back-fill logic: Older posts that were published before the media_id key was introduced stored only a permalink in the result key. Both the check-live and archive routes include a back-fill path: they extract the numeric ID from instagram.com/p/{numeric_id}/ in the result URL and persist it to the media_id key for future calls.

Archive Post

The POST /admin/articles/<id>/archive-ig-post route clears the current post's active keys and appends a history entry. The history is persisted in ig_car_post_history_{id} and grows indefinitely (it is never cleared or capped). Each history entry contains: 

{
    "media_id":    "17854360229135492",
    "permalink":   "https://www.instagram.com/p/CxYZ1234567/",
    "archived_at": "2026-03-14T11:00:00Z",
    "caption":     "Caption text from snapshot…",
    "image_urls":  ["https://storage.googleapis.com/…"],
    "video_urls":  [],
    "post_type":   "car",
    "posted_at":   "2026-03-14T10:30:00Z"
}

After archiving, the following keys are set to empty strings (not deleted): ig_car_post_result_{id}, ig_car_post_media_id_{id}, ig_car_post_snapshot_{id}. The status key is not touched (it will be absent if no worker is running). The frontend resets its UI to the "not posted" state after a successful archive.

7. Activity Log

Every significant event for an article is recorded in a per-article activity log stored in site_settings under the key ig_activity_log_{article_id}. The log is shared across all components (carousel image generation, Instagram posting, cinemagraph generation, narrated video, YouTube upload).

_add_activity_log(article_id, title, content, component=None)

Appends a single entry to the log. Reads the existing JSON array, appends the new entry, caps the array at 50 entries (oldest discarded first), and upserts the full JSON back into site_settings. 

def _add_activity_log(article_id, title, content, component=None):
    key = f"ig_activity_log_{article_id}"
    raw = get_setting(key)
    entries = json.loads(raw) if raw else []
    if len(content) > 4000:
        content = content[:4000] + "\n… (truncated)"
    entry = {
        "title":     title,
        "timestamp": datetime.utcnow().strftime("%Y-%m-%dT%H:%M:%SZ"),
        "content":   content,
    }
    if component:
        entry["component"] = component
    entries.append(entry)
    entries = entries[-50:]  # keep last 50
    val = json.dumps(entries)
    execute(
        "INSERT INTO site_settings (`key`, value) VALUES (%s, %s) "
        "ON DUPLICATE KEY UPDATE value = %s",
        (key, val, val),
    )

SSE Streaming Pattern

Because EventSource only supports GET, the frontend uses fetch() with an AbortController and reads the response body as a ReadableStream: 

carouselAbortCtrl = new AbortController();

fetch('/admin/articles/' + ARTICLE_ID + '/generate-carousel', {
    method: 'POST',
    signal: carouselAbortCtrl.signal
})
.then(res => {
    if (!res.ok) return res.json().then(d => { throw new Error(d.error || 'Server error'); });
    const reader = res.body.getReader();
    const dec = new TextDecoder();
    let buf = '';

    function read({ done, value }) {
        if (done) { carouselDone(-1); return; }
        buf += dec.decode(value, { stream: true });
        const parts = buf.split('\n\n');     // split on SSE event boundaries
        buf = parts.pop();                   // keep partial last chunk in buffer
        parts.forEach(part => {
            let ev = 'message', data = '';
            part.split('\n').forEach(line => {
                if (line.startsWith('event: ')) ev = line.slice(7).trim();
                if (line.startsWith('data: '))  data = line.slice(6).trim();
            });
            if (!data) return;
            try { handleCarouselEvent(ev, JSON.parse(data)); } catch(e) {}
        });
        reader.read().then(read);
    }
    reader.read().then(read);
})
.catch(err => {
    if (err.name === 'AbortError') { carouselStopped(); }
    else { /* show error, re-enable buttons */ }
});
IG posting uses polling, not SSE. The post-to-instagram endpoint returns immediately after spawning the background thread. Progress is tracked by polling ig-post-status?type=car every 5 seconds. This is the correct approach for long-running operations that outlast a typical HTTP request timeout.

10. End-to-End Logic Flows

A. Full Generation (all 10 slots empty)

  1. User clicks "Generate Carousel" button (or kebab menu item).
  2. startCarousel(): disables Generate buttons, shows Stop + progress bar. Adds placeholders cph-1 through cph-10 to the current grid.
  3. Frontend POSTs to /admin/articles/{id}/generate-carousel.
  4. Backend: reads active_style from site_settings. Loads existing arrays from DB (all null / empty → empty_slots = [0,1,...,9], N = 10).
  5. Backend: GPT-4.1-mini call — system prompt instructs it to generate exactly 10 {prompt, punchline} objects as a JSON array.
  6. Backend: yields event: prompts. Frontend updates status to "Generating images…".
  7. Backend: for each of the 10 slots — calls DALL-E 3 with "{prompt}. {active_style}"; downloads the image bytes; saves to local disk as static/articles/{id}/carousel/image_{N}.png; uploads to GCS via upload_bytes_to_gcs().
  8. Backend: yields event: image {index, seq, of=10, url, prompt, punchline} where url is the GCS public URL (or local static URL if GCS not configured). Frontend: replaces placeholder with image tile, advances progress bar.
  9. After all slots: backend yields event: done {count}.
  10. Backend: UPDATE articles — writes carousel_prompts, carousel_punchlines, carousel_images (GCS URLs), carousel_style. Writes to activity log.
  11. Frontend: carouselDone(count) — cleans up, re-enables buttons, calls updateCarouselTabCounts() which hides Generate button (all 10 slots now filled).

B. Partial Generation (some slots already filled)

  1. Same as Full Generation steps 1–3.
  2. Backend detects N < 10 empty slots. GPT is asked for exactly N scenes.
  3. Images are generated only for the N empty slot indices; existing images in filled slots are untouched.
  4. SSE image events: index = the actual slot number (1–10), seq = position in current generation (1–N), of = N.
  5. Frontend progress: (seq / of) × 100%. Status: "Generated {seq} of {N}…".
  6. DB save: merged_* arrays preserve existing filled slots; new values go into the correct empty indices.

C. Per-Image Re-run

  1. User clicks "i" button on a current image tile. openCarouselEditModal(index0, prompt, punchline, false) — modal opens in edit mode with pre-filled textareas.
  2. User edits prompt and/or punchline and clicks "Re-run this prompt".
  3. rerunCarouselImage() POSTs {index: 0–9, prompt, punchline} to regenerate-carousel-image.
  4. Backend validates index is 0–9. Reads existing arrays. Saves the old prompt/image/punchline at that index.
  5. Backend calls DALL-E 3. Saves new image as image_{index+1}_{timestamp}.png (timestamped to preserve old file). Uploads to GCS.
  6. Backend: updates the slot with new values (GCS URL if upload succeeded); appends old values to the end of the arrays (slot 10+). Saves to DB. Writes to activity log.
  7. Frontend: addRemovedImage(oldTile) moves the old tile to the Removed grid. addCarouselImage() adds the new tile. Modal closes. updateCarouselTabCounts() runs.

D. Archive All

  1. User clicks "Archive All" button.
  2. archiveAllCarousel() POSTs to archive-carousel.
  3. Backend: collects non-null items from slots 0–9 as new_archived. Appends after any existing archived items. Resets slots 0–9 to None. Saves.
  4. Frontend: all cimg-* tiles are moved to removed grid via addRemovedImage(). Current grid is now empty. updateCarouselTabCounts(): Generate button becomes visible.

E. Style Change

  1. Admin navigates to /admin/settings. Settings tab shows the current site_settings.carousel_style in a textarea.
  2. Admin edits text and clicks "Save Settings".
  3. saveSettings() POSTs {carousel_style: "..."} to /admin/settings/save.
  4. Backend: upserts via INSERT ... ON DUPLICATE KEY UPDATE. Returns {success: true}.
  5. From the next carousel generation onwards, the new style is used. The article's carousel_style column is updated to record which style produced its images.
  6. Existing articles' "View Style" modals show their own saved carousel_style value, not the new site setting.

F. Instagram Posting (carousel)

  1. User optionally clicks "Generate Caption" — generateIgCaption() POSTs to generate-ig-caption with {component: "carousel"}. GPT-4.1-mini returns a caption (up to 600 tokens); the frontend populates the caption textarea.
  2. User reviews/edits the caption (up to 2200 characters) and clicks "Post to Instagram".
  3. postToInstagram() POSTs {caption, type: "car"} to post-to-instagram. Backend validates credentials, checks for in-progress worker, validates image count (≥2), seeds ig_car_post_status_{id} = "running:starting", spawns daemon thread, returns {started: true}.
  4. Frontend disables Post button and starts polling loop (every 5 seconds).
  5. Worker — compose phase (running:compose:{i}/{n}): for each carousel image, download bytes (from GCS URL or local path), apply punchline overlay via compose_carousel_slide(), encode as JPEG, upload composed JPEG to GCS as articles/{id}/instagram/composed_{i}_{ts}.jpg. Fails immediately if any upload fails (GCS required).
  6. Worker — child containers (running:containers:{i}/{n}): POST each composed GCS URL to the Instagram Graph API as a carousel item (is_carousel_item=true). Fails immediately if any container creation fails.
  7. Worker — poll children (running:poll:{i}/{n}): poll each child container's status_code until FINISHED or ERROR. Polls with exponential back-off up to 30s; 5-minute timeout per container.
  8. Worker — carousel container (running:carousel): POST media_type=CAROUSEL with all child IDs and the caption.
  9. Worker — poll carousel (running:carousel_poll): poll the carousel container until FINISHED. Same timeout and back-off as child polling.
  10. Worker — publish (running:publish): POST to /{ig_user_id}/media_publish with the carousel container ID.
  11. Worker — verify (running:publish_verify, conditional): if publish returns an error, wait 10s and check the account's most recent 3 posts. If any post is less than 2 minutes old, treat it as a successful publish despite the error response (Instagram sometimes returns rate-limit/Fatal but publishes anyway).
  12. Worker — success path: fetch the real permalink via GET /{post_id}?fields=permalink. Write ig_car_post_result_{id} = "done:{permalink}". Save media_id. Save post snapshot JSON. Write to activity log. Delete status key.
  13. Frontend polling detects non-running status → calls showIgSuccess(permalink, media_id). UI shows "View on Instagram" link, "Check if live", and "Archive post" buttons.

G. Archive Instagram Post

  1. User clicks "Archive post" button (visible after a done or warn result).
  2. archiveIgPost() POSTs {type: "car"} to archive-ig-post.
  3. Backend reads media_id, result (for permalink), and snapshot. Appends history entry with all fields. Persists updated history to ig_car_post_history_{id}. Sets result, media_id, and snapshot keys to empty strings. Writes to activity log.
  4. Frontend: hides result panel, re-shows caption textarea and Post button. UI is ready for a new post.

11. Slot Index Convention

All three carousel arrays (carousel_prompts, carousel_images, carousel_punchlines) share the same index convention:

Index rangeMeaningUI location
0–9Active / current carousel slots"Current" tab in carousel editor
10+Archived / removed items"Removed" tab in carousel editor

A null / None entry in slots 0–9 means that slot is empty. The Generate button and kebab menu item are enabled only when 10 - currentCount > 0, i.e., at least one slot is empty.

Array padding: Before computing empty_slots, the existing arrays are padded to exactly 10 entries: (existing[:10] + [None] * 10)[:10]. This ensures empty_slots is always computed against a 10-element list, even if the DB array is shorter (e.g., a partial generation that stopped at 5).

Archive boundary preservation

The archive-carousel route writes a [None] * 10 prefix before the archived items. This ensures that existing[10:] in generate-carousel always correctly slices only truly archived items, regardless of how many were archived: 

# After archive-carousel, arrays always look like:
final_p  = [None] * 10 + already_archived_p  + new_archived_p
final_i  = [None] * 10 + already_archived_i  + new_archived_i
final_pl = [None] * 10 + already_archived_pl + new_archived_pl
# So existing[10:] == already_archived + new_archived  ✓

IG posting uses only active slots

The Instagram posting worker reads carousel_images[:10] and filters out None values: [x for x in images[:10] if x]. Only non-null images from the active slots are posted. Archived images (slots 10+) are never included.
