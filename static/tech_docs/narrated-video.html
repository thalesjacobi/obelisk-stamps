Narrated Video Generator

Project: Obelisk Stamps Component: B — article_edit.html Date: March 2026

1. Overview

The Narrated Video Generator is the most complex admin component in the Obelisk Stamps application. It converts an article into a fully narrated MP4 video suitable for Instagram Reels and YouTube Shorts, with no manual editing required. The entire pipeline runs in a background thread and produces a publication-ready video in 5–15 minutes.

High-level pipeline

  1. Script: GPT-4.1-mini reads the article and writes exactly 10 narration segments.
  2. TTS: OpenAI Text-to-Speech converts each segment to an audio file.
  3. Frames: FFmpeg renders a video clip per segment — either a Ken Burns pan/zoom over a carousel image, or a looping cinemagraph clip.
  4. Mux: Each segment's video and audio are combined into one timed clip.
  5. Concat: All 10 clips are joined into a single continuous video.
  6. Speed-up: The full video is sped up 1.35× using FFmpeg setpts + atempo.
  7. Upload: The final MP4 is uploaded to Google Cloud Storage and the run is recorded in the DB.

3. Database Schema

articles table — narrated video columns

ColumnTypeDescription
video_narrated_url TEXT URL of the most recent generated video. Legacy field — superseded by the url field inside video_narrated_runs. Kept for backwards compatibility.
video_narrated_script TEXT JSON array of narration segment strings from the latest run. Populated during Step 1 of the worker so it is available for display even before rendering completes.
video_narrated_runs TEXT JSON array of run objects (see schema below). Newest run is prepended to the front of the array. Never overwritten — each generation adds one entry.
video_narrated_status TEXT Current generation status string. NULL when idle. Set to a running:… or error:… string while the worker is active. Reset to NULL on completion or cancellation.
video_narrated_log TEXT Live generation log. Accumulated during worker execution — includes FFmpeg output, step transitions, and error traces. Flushed periodically rather than per-line for DB efficiency. Cleared at the start of each new generation run.
Status vs. NULL: The worker sets video_narrated_status = NULL (not an empty string) on successful completion or clean cancellation. The frontend checks status == null to determine the idle state. Any non-null value means a run is in progress or has errored.

Run object schema

Each element of the video_narrated_runs JSON array has the following structure: 

{
  "ts":              1741993020,          // Unix timestamp (int) — used as unique run ID
  "ts_label":        "14 March 2026 21:17",
  "url":             "https://storage.googleapis.com/bucket/articles/42/video/narrated_1741993020.mp4",
  "params": {
    "format":        "vertical",          // "vertical" | "square" | "landscape"
    "voice":         "onyx",              // OpenAI TTS voice name
    "tts_model":     "tts-1",             // "tts-1" | "tts-1-hd"
    "script_len":    "medium",            // "short" | "medium" | "long"
    "kb_speed":      "medium",            // Ken Burns speed: "slow" | "medium" | "fast"
    "crf":           23,                  // FFmpeg CRF, 18–28
    "fps":           25,                  // Output FPS, 25 or 30
    "render_fps":    12,                  // Internal render FPS (fixed)
    "speed":         1.35                 // Speed-up factor (fixed)
  },
  "segments":        ["Segment 1 text…", "Segment 2 text…", /* … 10 total */],
  "cinemagraph_slides": [null, null, "https://…clip2.mp4", null, /* … 10 total */]
                         // Array of 10 elements — GCS URL of cinemagraph clip used for that
                         // slot, or null if Ken Burns was used instead
}
cinemagraph_slides: This array always has exactly 10 elements (one per segment slot). A non-null element means a cinemagraph clip was available and used for that slot during rendering. The array is preserved for auditing and for potential future re-render support.

4. Generation Parameters

These parameters are submitted in the POST body of the generate route. They are stored inside the run object under params for provenance tracking.

ParameterTypeValuesDefaultDescription
format string vertical, square, landscape vertical Output video dimensions. Vertical = 720×1280 (9:16, ideal for Reels/Shorts). Square = 1080×1080 (1:1). Landscape = 1280×720 (16:9).
voice string onyx, nova, alloy, echo, fable, shimmer onyx OpenAI TTS voice. Each has a distinct character: onyx is deep/authoritative, nova is warm/conversational, alloy is neutral, echo is reserved, fable is expressive, shimmer is bright.
tts_model string tts-1, tts-1-hd tts-1 OpenAI TTS model quality tier. tts-1 is faster and cheaper (~$0.015/1K chars). tts-1-hd produces higher-fidelity audio (~$0.03/1K chars).
script_len string short, medium, long medium Target length of each segment. Short ≈ 3 sentences, medium ≈ 5, long ≈ 7. Controls the GPT-4.1-mini prompt instruction. Affects total video duration.
kb_speed string slow, medium, fast medium Ken Burns zoom/pan speed for still-image slides. Mapped to a zoom rate multiplier in the FFmpeg zoompan filter. Does not affect cinemagraph slides.
crf integer 18–28 23 FFmpeg Constant Rate Factor for H.264 encoding. Lower values = higher quality + larger file. 18 is near-visually-lossless; 28 is the practical maximum for acceptable quality. Each unit ≈ 12% file size change.
fps integer 25, 30 25 Output video frame rate. Affects smoothness of the final video. 25 fps is standard for PAL/European formats; 30 fps for NTSC/US formats.
render_fps integer 12 (fixed) 12 Internal rendering frame rate used for the Ken Burns frame generation step. Fixed at 12 to significantly reduce FFmpeg processing time. The output is interpolated to the target fps during mux.
speed float 1.35 (fixed) 1.35 Post-processing speed-up factor applied to the final concatenated video. Implemented via FFmpeg setpts=PTS/1.35 (video) and atempo=1.35 (audio). Reduces total duration by ~26% and tightens pacing.
Duration estimate: Each of the 10 segments will have TTS audio of roughly 8–24 seconds depending on script_len. Total pre-speed-up duration is approximately 80–240 seconds. After the 1.35× speed-up, final duration is approximately 60–180 seconds (1–3 minutes).

6. Backend Routes

Generation and status routes

POST/admin/articles/<id>/generate-narrated-video

Starts a new narrated video generation run. Validates that no run is already in progress, then spawns _narrated_video_worker in a daemon thread.

Request body (JSON):

{
  "format":     "vertical",   // required
  "voice":      "onyx",       // required
  "tts_model":  "tts-1",      // required
  "script_len": "medium",     // required
  "kb_speed":   "medium",     // required
  "crf":        23,           // required, integer 18–28
  "fps":        25            // required, 25 or 30

{ "ok": true }                       // generation started
{ "error": "already running" }       // if status is non-null (concurrent guard)

Response (JSON):

7. Worker Pipeline

The worker function _narrated_video_worker(article_id, params) runs in a daemon thread. It writes status updates to video_narrated_status between each step so the frontend can display a progress bar. All temporary files are created in the system temp directory and cleaned up after a successful GCS upload.

1

Script generation — running:script

The article's title and HTML content are fetched from the DB. The HTML is stripped of tags and truncated to 8,000 characters to stay within GPT context limits.

A GPT-4.1-mini chat completion is requested with a system prompt that instructs it to produce exactly 10 narration segments. The script_len parameter maps to a target-sentences instruction in the prompt: short ≈ 3 sentences, medium ≈ 5, long ≈ 7.

The response is parsed into a list of 10 strings and written to video_narrated_script.

# Example status during this step
video_narrated_status = "running:script"

# On completion
video_narrated_script = '["Welcome to our deep dive into…", "The first issue…", …]'
2

TTS per segment — running:tts:N/10

For each of the 10 segments (N = 1 through 10), the OpenAI TTS API is called with:

  • model = params["tts_model"] (tts-1 or tts-1-hd)
  • voice = params["voice"]
  • input = segment text string

The response audio is written to a temporary MP3 file. The exact duration of the audio file is then determined by running ffprobe and parsing the duration field from the format metadata. This duration drives the video clip length in the next step. 

# Status format (N is 1-indexed)
video_narrated_status = "running:tts:3/10"
3

Frame rendering per segment — running:clip:N/10

For each segment, the worker selects a visual source and renders a silent video clip of the exact duration determined in Step 2.

Source selection logic

The worker checks cinemagraph_slides[N] from the article's cinemagraph data. If a GCS URL is present for slot N, the cinemagraph branch is used. Otherwise, the Ken Burns branch is used with the carousel image for slot N (or the last available carousel image if slot N doesn't exist).

Cinemagraph branch

The cinemagraph clip (a looping MP4 from GCS) is used as input with infinite loop enabled:

ffmpeg -stream_loop -1 -i cinemagraph.mp4 \
  -vf "scale=S:S,crop=W:H,fps=RENDER_FPS" \
  -t DURATION \
  -an segment_N_video.mp4

Where S×S is a square intermediate size, and W×H is the target output resolution for the selected format. The fps filter is applied before the scale filter to limit the number of frames being scaled.

Ken Burns branch

A still image is animated with a slow zoom-and-pan (Ken Burns effect) using the FFmpeg zoompan filter:

ffmpeg -loop 1 -i carousel_image.jpg \
  -vf "fps=RENDER_FPS,scale=W*2:H*2,zoompan=z='min(zoom+SPEED,1.5)':d=FRAMES:s=WxH" \
  -t DURATION \
  -an segment_N_video.mp4

Key ordering: fps filter is applied before scale to reduce the number of frames that must pass through the expensive scale operation. RENDER_FPS = 12 (fixed), so the filter chain processes only 12 frames per second regardless of the final output FPS.

SPEED is the Ken Burns zoom increment per frame, derived from kb_speed:

kb_speedZoom increment per frame
slow0.0005
medium0.001
fast0.002
# Status format
video_narrated_status = "running:clip:5/10"
4

Mux audio + video per segment

Each silent segment video (from Step 3) is combined with its TTS audio (from Step 2) using FFmpeg's -shortest flag to ensure they are exactly duration-matched: 

ffmpeg -i segment_N_video.mp4 -i segment_N_audio.mp3 \
  -c:v copy -c:a aac \
  -shortest \
  segment_N_muxed.mp4

The video stream is copied without re-encoding (-c:v copy) to avoid quality loss and speed up this step. Audio is encoded to AAC for compatibility.

5

Concatenate all segments — running:merge

A concat.txt file is written to the temp directory listing all 10 muxed segment paths in order: 

file '/tmp/narrated_42_1741993020/segment_0_muxed.mp4'
file '/tmp/narrated_42_1741993020/segment_1_muxed.mp4'
...
file '/tmp/narrated_42_1741993020/segment_9_muxed.mp4'

FFmpeg's concat demuxer joins them into a single MP4:

ffmpeg -f concat -safe 0 -i concat.txt \
  -c copy \
  full_video.mp4

Stream copy is used again — no re-encoding occurs at this step.

6

Speed-up 1.35× — running:speed

The concatenated video is sped up using two FFmpeg filters applied simultaneously: 

ffmpeg -i full_video.mp4 \
  -vf "setpts=PTS/1.35" \
  -af "atempo=1.35" \
  -c:v libx264 -crf CRF -preset medium \
  -c:a aac \
  -r FPS \
  final_sped_up.mp4

setpts=PTS/1.35 speeds up the video stream by manipulating presentation timestamps. atempo=1.35 speeds up audio while preserving pitch (FFmpeg's built-in time-scale modification). The video must be re-encoded at this step because PTS manipulation requires a full encode pass. CRF and FPS from the user's parameters are applied here.

After this step, all temporary files (per-segment video, audio, and muxed clips) are deleted.

7

Upload to GCS — running:upload

The final MP4 is uploaded to GCS using the shared upload_bytes_to_gcs() helper. The GCS object path is: 

articles/{article_id}/video/narrated_{ts}.mp4

After successful upload, a run object is constructed (see schema in Section 3) and prepended to the video_narrated_runs JSON array in the DB. The video_narrated_url column is also updated to the new URL for backwards compatibility. The status is set to NULL to signal completion.

The local temp MP4 file is deleted after upload.

running:tts:1/10 … running:tts:10/10TTS synthesis for segment N10%–40%

8. Instagram Reels Flow

The function _post_narrated_reel_worker(article_id, video_url, caption, run_ts) runs in a daemon thread and implements the full Instagram Graph API Reels publishing sequence.

  1. Create REELS container: POST to https://graph.facebook.com/v19.0/{ig_user_id}/media with:
    • media_type=REELS
    • video_url — must be the public GCS URL (Instagram fetches the video server-side)
    • caption — the user-provided caption text
    • access_token — from IG_ACCESS_TOKEN env var
    Returns a creation_id.
  2. Poll container status: GET https://graph.facebook.com/v19.0/{creation_id}?fields=status_code repeatedly with exponential back-off (initial 5s, max 30s). Expected progression: EXPIRED → IN_PROGRESS → FINISHED. If status_code = ERROR, log and abort.
  3. Publish: POST to https://graph.facebook.com/v19.0/{ig_user_id}/media_publish with creation_id and access_token. Returns a media_id.
  4. Verify (on publish error): If the publish call returns an error, wait 10 seconds then GET /{ig_user_id}/media?limit=3 to check if the post actually went through (Instagram sometimes errors even on success). If the most recent post matches, treat as success.
  5. Fetch permalink: GET https://graph.facebook.com/v19.0/{media_id}?fields=permalink to obtain the public Instagram URL.
  6. Persist result: Write narrated_ig_result_{id}_{ts} = "done:https://www.instagram.com/p/…" and narrated_ig_media_id_{id}_{ts} = media_id to site_settings. Append an entry to ig_activity_log_{article_id} with component="narrated".
Video URL requirement: Instagram's server must be able to fetch the video URL directly. GCS URLs work as long as the bucket objects are publicly readable. Private or signed URLs will cause the container to enter ERROR status.

9. YouTube Shorts Flow

The function _upload_to_youtube_worker(article_id, video_url, title, description, run_ts) runs in a daemon thread and implements the YouTube Data API v3 resumable upload flow.

  1. Refresh access token: POST to https://oauth2.googleapis.com/token with:
    • client_id, client_secret from env vars
    • refresh_token from site_settings["youtube_refresh_token"]
    • grant_type=refresh_token
    Returns a short-lived access_token.
  2. Download video to temp file: Stream the GCS video URL to a local temp file using requests.get(video_url, stream=True). Required because the YouTube API expects a seekable file object for resumable uploads.
  3. Initiate resumable upload: POST to https://www.googleapis.com/upload/youtube/v3/videos?uploadType=resumable&part=snippet,status with JSON body: 
    {
  "snippet": {
    "title":       "…",
    "description": "…",
    "tags":        ["stamps", "philately", "obelisk"],
    "categoryId":  "26"    // Howto & Style
  },
  "status": {
    "privacyStatus": "public"
  }
}
    Additional headers: X-Upload-Content-Type: video/mp4, X-Upload-Content-Length: {file_size}. Response header Location contains the resumable upload URL.
  4. Upload file: PUT to the resumable upload URL with the file bytes. Response body contains the created video resource including id (the video ID).
  5. Persist result: Write yt_result_{id}_{ts} = "done:https://www.youtube.com/shorts/{video_id}" to site_settings. Delete the local temp file.

YouTube OAuth flow

  1. /admin/youtube-connect: Build authorization URL with scope https://www.googleapis.com/auth/youtube.upload and access_type=offline (required for refresh token). Redirect browser to Google's consent page.
  2. User grants consent: Google redirects to {SITE_URL}/admin/youtube-oauth-callback?code=…
  3. /admin/youtube-oauth-callback: Exchange the code for tokens via POST to oauth2.googleapis.com/token. Extract and store the refresh_token in site_settings. The refresh token persists until revoked — the OAuth flow only needs to be run once per YouTube account.
YouTube quota: The YouTube Data API v3 has a daily quota of 10,000 units. A single video upload costs approximately 1,600 units, allowing roughly 6 uploads per day before the quota is exhausted. Quota resets at midnight Pacific Time.

11. Frontend JavaScript

All JavaScript for this component is embedded in a <script> block at the bottom of article_edit.html. It uses no external JS libraries beyond Bootstrap 5.

Status polling

When generation is active, the frontend polls /admin/articles/{id}/narrated-video-status every 2 seconds. Polling is started by the generate button click handler and stopped when status === null (idle) or status.startsWith("error"). 

let _narratedPollTimer = null;

function _narratedStartPolling() {
  _narratedPollTimer = setInterval(async () => {
    const data = await fetch(`/admin/articles/${ARTICLE_ID}/narrated-video-status`).then(r => r.json());
    _narratedApplyStatus(data.status, data.runs, data.script);
    if (data.status === null || data.status.startsWith("error")) {
      clearInterval(_narratedPollTimer);
      _narratedPollTimer = null;
      _narratedSetIdle(data.status === null);
    }
  }, 2000);
}

_narratedApplyStatus(status, runs, script)

Maps the status string to a human-readable label and a progress percentage, then updates the progress bar and status label in the DOM. 

function _narratedApplyStatus(status, runs, script) {
  if (!status) return;

  let label = status;
  let pct   = 0;

  if (status === "running:script") {
    label = "Generating narration script…"; pct = 5;

  } else if (status.startsWith("running:tts:")) {
    const n = parseInt(status.split(":")[2]);         // e.g. "3/10" → 3
    const num = parseInt(n.toString().split("/")[0]);
    label = `TTS synthesis — segment ${n}…`;
    pct = 10 + (num / 10) * 30;   // 10% → 40%

  } else if (status.startsWith("running:clip:")) {
    const n   = status.split(":")[2];
    const num = parseInt(n.split("/")[0]);
    label = `Rendering video clip — segment ${n}…`;
    pct = 40 + (num / 10) * 35;   // 40% → 75%

  } else if (status === "running:merge") {
    label = "Merging segments…"; pct = 80;

  } else if (status === "running:speed") {
    label = "Applying 1.35× speed-up…"; pct = 90;

  } else if (status === "running:upload") {
    label = "Uploading to GCS…"; pct = 95;

  } else if (status.startsWith("error:")) {
    label = "Error: " + status.slice(6); pct = 0;
  }

  document.getElementById("narrated-progress-bar").style.width = pct + "%";
  document.getElementById("narrated-status-label").textContent = label;
}

Instagram JavaScript

Each accordion item's Instagram section is wired with event listeners after the item is inserted into the DOM. Key behaviours:

  • Caption character counter: input event on the textarea updates the counter span. Max 2,200 characters enforced by maxlength attribute.
  • Generate caption link: click → POST /generate-ig-caption with {component: "narrated"}. Response caption is inserted into the textarea and counter updated.
  • Post to Instagram button: Reads data-video-url and data-run-ts from the button's dataset. Disables button, shows progress row, POSTs to /post-narrated-to-instagram, then starts polling /narrated-ig-status?ts={run_ts} every 3 seconds.
  • IG status polling: Stops when result is non-null. If result.startsWith("done:"), shows success row with permalink link. If result.startsWith("error:"), shows error message.
  • Check if post is live: POSTs to /check-narrated-post-live and updates the button label with the result.
  • Archive post: POSTs to /archive-narrated-post with confirmation dialog first.

YouTube JavaScript

  • Post to YouTube Shorts button: Reads data-video-url and data-run-ts. Collects title and description from the sibling inputs. Disables button, shows progress row, POSTs to /post-to-youtube.
  • YT status polling: Polls /youtube-status?ts={run_ts} every 3 seconds. Stops when result is non-null. On done: result, extracts the YouTube URL and shows the success link.
  • Title pre-fill: The title input is pre-filled from the article title (injected by Jinja2 template). The user may edit it before posting.
Narration script display: Each accordion item includes a collapsible section showing the 10 narration segments as text. This section is collapsed by default. It is populated from run.segments inside buildRunItem().

12. Logic Flows

A. Start generation

  1. User configures parameters (format, voice, tts_model, script_len, kb_speed, crf, fps) in the controls panel.
  2. User clicks Generate Narrated Video.
  3. JS disables the generate button and all parameter inputs, shows the progress bar row.
  4. JS POSTs to /generate-narrated-video with parameter JSON.
  5. If response is {"error": "already running"}, JS shows an alert and re-enables controls.
  6. If response is {"ok": true}, JS starts the 2-second polling loop.
  7. Each poll updates the progress bar and status label via _narratedApplyStatus().
  8. When polling detects status === null, generation is complete:
    1. JS stops polling.
    2. JS calls buildRunItem(runs[0]) for the newest run.
    3. New accordion item is prepended to the runs accordion.
    4. Accordion item is automatically expanded.
    5. Progress bar row is hidden; controls are re-enabled.
  9. If polling detects status.startsWith("error:"):
    1. JS stops polling.
    2. Progress bar turns red; status label shows error message.
    3. Controls are re-enabled.

B. Cancel generation

  1. User clicks × (cancel) button in the progress bar row.
  2. JS POSTs to /cancel-narrated-video.
  3. Server writes cancelled to video_narrated_status.
  4. Worker detects cancelled status at next step boundary, exits cleanly.
  5. Worker sets status to NULL on exit.
  6. Polling detects null and re-enables controls.
  7. Any partially created temp files are left in the temp directory (cleaned at next generation start).

C. Post to Instagram Reels

  1. User opens a run's accordion item.
  2. (Optional) User clicks Generate caption and edits the auto-generated text.
  3. User clicks Post to Instagram Reels.
  4. JS disables the post button, shows the IG progress row.
  5. JS POSTs {video_url, caption, run_ts} to /post-narrated-to-instagram.
  6. JS starts polling /narrated-ig-status?ts={run_ts} every 3 seconds.
  7. Progress bar animates while polling returns non-final status strings.
  8. When result is non-null:
    • If done: — progress row hidden, success row shown with permalink link.
    • If error: — progress row shows error in red.
  9. (Optional) User clicks Check if post is live to verify public visibility.
  10. (Optional) User clicks Archive post (with confirmation) to hide from profile.

D. Upload to YouTube Shorts

  1. If YouTube is not connected, user clicks Connect YouTube account link.
  2. Browser is redirected to /admin/youtube-connect.
  3. Server redirects to Google's OAuth consent page.
  4. User grants youtube.upload permission.
  5. Google redirects to /admin/youtube-oauth-callback with authorization code.
  6. Server exchanges code for refresh token, stores it, redirects back to article edit page.
  7. User opens a run's accordion item, edits title/description as desired.
  8. User clicks Post to YouTube Shorts.
  9. JS disables button, shows YT progress row.
  10. JS POSTs {video_url, title, description, run_ts} to /post-to-youtube.
  11. JS starts polling /youtube-status?ts={run_ts} every 3 seconds.
  12. When result is done:https://…: success row shown with YouTube Shorts link.

YouTube Shorts duration: YouTube classifies vertical videos under 60 seconds as Shorts. Longer videos are uploaded as standard YouTube videos (not Shorts), even if the URL contains /shorts/. To reliably target Shorts, use script_len=short to keep total duration under 60 seconds post-speed-up.
Performance tip: The render_fps=12 fixed setting is a deliberate trade-off. Ken Burns zoompan at 25–30 fps is extremely CPU-intensive because FFmpeg must process a full frame for every output frame. Reducing to 12 fps for the render pass and upsampling during the final encode reduces total rendering time by approximately 50–60% with minimal perceptible quality loss in the final 1.35×-sped-up output.
