Cinemagraph Generator

Project: Obelisk Stamps Component: D — article_edit.html Date: March 2026

1. Overview

The Cinemagraph Generator is Component D of the article edit admin page. It animates each carousel image into a 5-second looping MP4 video clip using Luma AI Ray-2 image-to-video generation. Up to 10 clips can be generated, one per carousel slide.

The generated clips serve two downstream purposes:

  1. Instagram carousel posts — each clip becomes a video slide in a multi-video carousel posted via the Instagram Graph API.
  2. Narrated Video (Component E) — clips replace the static Ken Burns effect as the visual layer when a narrated video is assembled via FFmpeg.

Key capabilities

CapabilityDescription
Slot-based storage10 active slots (indices 0–9) backed by a JSON array column. Each slot maps 1:1 to its carousel image. null = slot is empty and eligible for generation.
Partial generationOnly empty slots are generated. Existing clips in filled slots are untouched and preserved in the DB merge.
Per-slide re-runAny individual clip can be replaced with a new Luma generation using a custom per-slide prompt. The old clip is moved to the Archived tab.
Re-run AllBulk re-generation with an optional global prompt override. Archives all existing clips then regenerates all 10 (or however many carousel images exist).
Archive / RestoreClips can be individually archived out of an active slot. Archived clips can be restored back into any empty or occupied slot.
Cancel supportUser can cancel a running generation. The worker checks a site_settings cancel key between each slide and stops cleanly.
Default promptA per-article prompt override can be saved (stored in site_settings). Falls back to a site-wide global default.
Polling UINo SSE stream — the frontend polls /video-data every 20 seconds and renders newly completed clips as they arrive.
Log captureThe worker writes timestamped log lines to carousel_cinemagraph_log (flushed periodically). Viewable from a "View Logs" button.
Prerequisite: Cinemagraph generation requires an existing carousel (Component B). The Generate Cinemagraph button is locked until at least one carousel_images slot is filled. The lock is lifted by cinemagraphUnlock(), which is called when carousel images are loaded.

3. Database Schema

Table: articles — cinemagraph columns

Five TEXT columns are added to the existing articles table. All are added via idempotent ALTER TABLE … ADD COLUMN calls at startup, wrapped in try/except to silently skip if the column already exists.

ColumnTypePurpose
carousel_cinemagraphs TEXT (JSON array) Array of 10 GCS or local URLs for active clip slots. null entries mean that slot is empty. Index 0 = slide 1.
carousel_cinemagraph_log TEXT Free-text generation log. Flushed periodically by the worker during a run. Shown in the "View Logs" modal. Capped at 65,000 characters to stay within MySQL TEXT limits.
carousel_cinemagraph_prompts TEXT (JSON array) Array of 10 per-slide Luma prompts. Written by the route before the worker starts, so the worker always reads its prompts from this array. Re-run also patches this column.
carousel_cinemagraph_archived TEXT (JSON array) Array of archived clip objects. Each object: {url, prompt, created_at, archived_at, slot_source}. See below.
carousel_cinemagraph_created_at TEXT (JSON array) Array of 10 ISO 8601 UTC timestamps (e.g. "2026-03-14T10:30:00Z") recording when each active clip was generated. null = no clip in that slot.
Slot convention: All five columns share the same index space. Indices 0–9 are the 10 active slots corresponding to carousel slides 1–10. The carousel_cinemagraph_archived column is a separate array (not indexed by slot) — it grows unboundedly as clips are archived.

Archived clip object schema

{
  "url":         "https://storage.googleapis.com/.../slide_3_1710000000.mp4",

  "archived_at": "2026-03-14T11:00:00Z",   // when it was moved to archive (ISO 8601)
  "slot_source": 3                          // 1-based slot it came from
}

site_settings keys

All transient state (running status, results, cancel signals) is stored in the site_settings key-value table rather than the articles table. This keeps the articles table clean and makes it safe to restart the server mid-run (stale cinemagraph_status_* keys are deleted at startup).

Key patternTypeLifecycleExample values
cinemagraph_status_{article_id} Transient string Written at run start; deleted when worker finishes or on server restart "running:0/10", "running:3/10:dreaming", "slide_running:2"
cinemagraph_result_{article_id} Transient string Written at run end; cleared by frontend via DELETE /cinemagraph-result after display "done", "error:Billing error: ...", "warn:partial:3/5"
cinemagraph_cancel_{article_id} Transient string Written by cancel route; deleted by worker when it detects cancellation "1"
cinemagraph_prompt_{article_id} Persistent string Written by "Save for this Article" button; persists indefinitely Custom Luma prompt text (any string)
cinemagraph_prompt Persistent string Site-wide default; seeded from _CINE_DEFAULT_PROMPT constant on first app start via INSERT IGNORE Default Luma motion prompt text
ig_post_status_{article_id} Transient string Written during IG post flow "running:start", "running:child_3", "running:publish"
ig_post_result_{article_id} Transient string Written when IG post completes "done:https://www.instagram.com/p/...", "error:..."
ig_post_snapshot_{article_id} Persistent JSON Written on successful IG post; shown in the "posted" state UI JSON object with post metadata
ig_activity_log_{article_id} Persistent JSON array Appended on each IG post action; shown in activity log modal Array of {type, message, url, timestamp} objects
Server restart cleanup: On app startup, init_site_settings() runs DELETE FROM site_settings WHERE key LIKE 'cinemagraph_status_%'. This prevents stale "running" status keys from blocking new generation requests after a crash or restart mid-run.

4. Python Backend (app.py)

Luma AI Client Initialisation

The Luma AI Python SDK (lumaai) is imported at module load. If the package is not installed or LUMAAI_API_KEY is not set, _luma_enabled remains False and every generation route returns a 400 error immediately. 

try:
    from lumaai import LumaAI as _LumaAI
except ImportError:
    _LumaAI = None

LUMAAI_API_KEY = os.getenv("LUMAAI_API_KEY", "")
_luma_client  = None
_luma_enabled = False

if LUMAAI_API_KEY and _LumaAI:
    try:
        _luma_client = _LumaAI(auth_token=LUMAAI_API_KEY)
        _luma_enabled = True
    except Exception as e:
        print(f"WARNING: Luma AI init failed: {e}")

Helper Functions

_luma_create_task(image_url, prompt_text) → str

Submits an image-to-video generation task to Luma. The source image must be a publicly accessible HTTPS URL — local static paths cannot be passed directly. Returns the Luma generation ID (UUID string). 

def _luma_create_task(image_url, prompt_text):
    gen = _luma_client.generations.create(
        prompt=prompt_text or "gentle subtle motion",
        model="ray-2",
        keyframes={"frame0": {"type": "image", "url": image_url}},
        aspect_ratio="1:1",
        duration="5s",
    )
    return gen.id

_luma_poll_task(gen_id) → (state, mp4_url, reason)

Fetches the current state of a generation. The tuple fields:

FieldTypeValues
statestr"queued", "dreaming", "completed", "failed"
mp4_urlstr | NoneCDN URL to the output MP4 (only when state == "completed")
reasonstrFailure reason string (only when state == "failed")
def _luma_poll_task(gen_id):
    gen = _luma_client.generations.get(id=gen_id)
    state   = gen.state   # "queued", "dreaming", "completed", "failed"
    mp4_url = gen.assets.video if state == "completed" and gen.assets else None
    reason  = gen.failure_reason if state == "failed" else ""
    return state, mp4_url, reason or ""

_is_luma_billing_error(exc) → bool

Detects non-transient errors that indicate the account has run out of credits. When this returns True, the worker aborts immediately and sets the result to "error:Billing/credits error: ..." rather than continuing to fail on every subsequent slide. 

def _is_luma_billing_error(exc):
    msg_lower = str(exc).lower()
    return any(phrase in msg_lower for phrase in [
        "insufficient", "credits", "billing", "quota", "balance",
        "permissiondenied", "403",
    ])

Route detail: POST generate-cinemagraphs

Request body (JSON, all fields optional):

{
  "prompts":       ["slide 1 prompt", null, "slide 3 prompt", ...],  // 10-element array, nulls use default
  "global_prompt": "optional override applied to all empty-prompt slots"
}

Pre-flight logic:

  1. Returns 400 if _luma_enabled is False.
  2. Returns 409 if a run is already in progress (cinemagraph_status_{id} starts with "running" or "slide_running").
  3. Loads carousel_images, pads to 10 slots.
  4. Resolves final prompt for all 10 slots: incoming per-slide prompt → global_prompt → article override → site default.
  5. Saves resolved prompts to carousel_cinemagraph_prompts.
  6. Computes empty_indices = slots that have a carousel image but no cinemagraph clip.
  7. Sets cinemagraph_status_{id} = "running:0/{N}", clears stale result key.
  8. Launches _cinemagraph_worker as a daemon thread.
  9. Returns {started: true, n_slides: N} immediately.

Route detail: GET video-data

The primary polling endpoint for the frontend. Returns a combined JSON object used by both the cinemagraph and narrated video components. Cinemagraph-relevant fields: 

{
  "cinemagraph_urls":       ["https://...", null, "https://...", ...],  // 10-slot array
  "cinemagraph_prompts":    ["prompt text", ...],
  "cinemagraph_archived":   [{"url": ..., "prompt": ..., ...}, ...],
  "cinemagraph_created_at": ["2026-03-14T10:00:00Z", null, ...],
  "cinemagraph_status":     "running:3/10:dreaming",  // or null if idle
  "cinemagraph_result":     null,                     // or "done" / "error:..." / "warn:partial:N/M"
  "has_cinemagraph_log":    true
}

Route detail: POST regenerate-cinemagraph-slide

Request body: {"slide_idx": 0–9, "prompt": "custom prompt text"}

  • Returns 409 if a run is already in progress.
  • Archives the current clip at slide_idx into carousel_cinemagraph_archived (returns the new archived_idx to the frontend).
  • Sets cinemagraph_status_{id} = "slide_running:{slide_idx}".
  • Launches _cinemagraph_slide_worker as a daemon thread.

Route detail: POST restore-cinemagraph-clip

Request body: {"archived_idx": N, "target_slot": 0–9}

  • Loads both carousel_cinemagraphs and carousel_cinemagraph_archived arrays.
  • If target_slot is already occupied, the displaced clip is appended to the archived array and returned as displaced: {entry, archived_idx} in the response.
  • The restored clip is removed from the archived array and written into the active slot.
  • Both arrays are saved back to the DB in one UPDATE.

5. Worker Detail — _cinemagraph_worker

Runs in a daemon thread started by admin_article_generate_cinemagraphs. Processes each empty slide in sequence (one at a time, not concurrent) so Luma rate limits are respected and cancel checks are reliable.

Arguments

def _cinemagraph_worker(
    article_id,
    images,          # list of source image URLs for the empty slots
    prompts=None,    # parallel list of per-slide prompts (same length as images)
    slot_indices=None,  # list mapping each item in images to its 0-based slot index
    existing_urls=None, # full 10-slot URL array — used to merge new results into
):
    ...
ParameterTypeNotes
article_idintDB article ID — used for status keys, log writes, DB updates
imageslist[str]Source image URLs to animate. Length = number of empty slots being processed.
promptslist[str]Parallel prompt list. Resolved before the worker starts by the route; each entry is a non-empty string.
slot_indiceslist[int]Maps each images[i] to its 0-based position in the 10-slot array. E.g. [0, 3, 7] means we are filling slots 1, 4, and 8.
existing_urlslist[str|None]Full current 10-slot URL array. Padded to 10 before use. New clip URLs are merged into this array; slots not being generated are preserved untouched.

Execution Flow

  1. Init: Pad existing_urls to 10. Load carousel_cinemagraph_created_at from DB and pad to 10. Delete any stale cancel key.
  2. For each (i, img_url) in enumerate(images):
    1. Check cancel key — break loop if set.
    2. Set cinemagraph_status_{id} = "running:{i}/{n}". Flush log.
    3. Resolve public URL: if the image is a GCS URL (https://), use it directly. If it's a local path, upload to articles/{id}/temp/slide_{N}_{ts}.jpg on GCS and use the resulting URL. Skip slide if upload fails or file not found.
    4. Call _luma_create_task(luma_img_url, prompt) → gen_id. If billing error, set billing_abort = True and break. Other errors: log and continue to next slide.
    5. Enter poll loop.
  3. Poll loop (see Luma Polling Pattern).
  4. On completed:
    1. Download MP4 from Luma CDN via requests.get(mp4_url, timeout=120).
    2. Save to static/articles/{id}/cinemagraph/slide_{N}_{ts}.mp4.
    3. Upload to GCS as articles/{id}/cinemagraph/slide_{N}_{ts}.mp4. If GCS succeeds, delete local file; otherwise use local path as fallback URL.
    4. Update cinemagraph_urls[real_idx] and cinemagraph_created_at[real_idx].
    5. Immediately UPDATE both columns to DB — the frontend poll will see the new clip on the next 20 s cycle.
  5. After all slides (or break):
    1. Delete cinemagraph_status_{id}.
    2. Set cinemagraph_result_{id}: "done" if all succeeded, "warn:partial:{M}/{N}" if some failed, "error:Billing..." if billing abort.
    3. Flush final log to DB.
    4. Upload log to GCS as articles/{id}/cinemagraph/run_{ts}.log (if GCS enabled).
    5. Optionally call activity log helper to record the completed run.

Luma Polling Pattern

The poll loop runs inside the per-slide processing. There is no maximum attempt count enforced in code, but Luma's typical generation time is 1–3 minutes. The poll interval is a fixed 6-second sleep (both in the full-run worker and the single-slide worker). 

mp4_url    = None
poll_count = 0
last_sub_status = None

while True:
    time.sleep(6)
    if _is_cancelled():
        _log(f"slide {slide_num} cancelled by user during poll")
        break
    poll_count += 1
    try:
        luma_status, mp4_url, status_msg = _luma_poll_task(gen_id)
    except Exception as e:
        _log(f"slide {slide_num} poll error: {e}")
        break
    if luma_status == "completed":
        break
    elif luma_status == "failed":
        _log(f"slide {slide_num} Luma FAILED (reason: {status_msg})")
        mp4_url = None
        break
    else:
        # "queued" or "dreaming" — keep polling
        if luma_status != last_sub_status:
            _set_status(f"running:{i}/{n}:{luma_status}")  # reflects sub-status in UI
            _flush_log()
            last_sub_status = luma_status
Sub-status propagation: When Luma's state changes from "queued" to "dreaming", the worker updates the status key to "running:3/10:dreaming". The frontend status-text renderer maps this to a human-readable label like "Dreaming (slide 4 / 10)".

Cancel Support

The route POST cancel-cinemagraphs writes cinemagraph_cancel_{id} = "1" to site_settings. The worker calls _is_cancelled() — which reads this key — at two points per slide:

  • Before submitting to Luma (between slides).
  • During the poll loop (inside the while True sleep cycle).

When cancellation is detected, the worker breaks out of the slide loop, clears the status key, and exits. The cancel key itself is deleted at the start of every new run to ensure a fresh state.

Log Flushing

The worker maintains an in-memory log_lines list. _flush_log() concatenates these lines with a run-start header and writes them to carousel_cinemagraph_log via a direct UPDATE. Flush is called:

  • At the start of each new slide (before Luma submission).
  • When Luma's sub-status changes (e.g. queued → dreaming).
  • At the end of the run (final flush before thread exits).

The log column is capped at 65,000 characters to stay safely within MySQL TEXT limits.

7. Instagram Posting Flow (Cinemagraph Carousel)

Posting a cinemagraph carousel to Instagram uses the same route as the static image carousel but with post_type: "cine" in the request body. The Instagram Graph API requires a multi-step container-based flow for video carousels.

IG Routes

MethodURLRequest body
POST /admin/articles/<id>/post-to-instagram {caption: "...", post_type: "cine"}
GET /admin/articles/<id>/ig-status?type=cine —
POST /admin/articles/<id>/check-instagram-post {post_type: "cine"}
POST /admin/articles/<id>/archive-ig-post {post_type: "cine"}
POST /admin/articles/<id>/generate-ig-caption {component: "cinemagraph"}

Posting Sequence

The backend IG worker runs in a daemon thread. Status is tracked via ig_post_status_{id} in site_settings.

  1. Collect clips: Load carousel_cinemagraphs (up to 10 non-null URLs). At least 2 clips required for a carousel; up to 10 supported (Instagram limit).
  2. Create child containers — one per clip: 
    POST https://graph.facebook.com/{ig_user_id}/media
  media_type=REELS
  video_url={clip_url}
  share_to_feed=false
  is_carousel_item=true
    Status: "running:child_0", "running:child_1", etc.
  3. Poll each child container for STATUS_CODE=FINISHED. Polls every ~5 s; times out after ~5 min per container.
  4. Create parent carousel container: 
    POST https://graph.facebook.com/{ig_user_id}/media
  media_type=CAROUSEL
  children={child_id_1},{child_id_2},...
  caption={caption}
    Status: "running:carousel"
  5. Poll parent container for FINISHED status. Status: "running:carousel_poll"
  6. Publish: 
    POST https://graph.facebook.com/{ig_user_id}/media_publish
  creation_id={parent_container_id}
    Status: "running:publish"
  7. Verify: If publish returns a Fatal error or rate-limit error, fall back to checking the account's recent media list for a newly created carousel post. Status: "running:publish_verify"
  8. Fetch permalink for the published post.
  9. Save snapshot to ig_post_snapshot_{id} in site_settings. Set ig_post_result_{id} = "done:https://www.instagram.com/p/...".
  10. Append activity log entry to ig_activity_log_{id}.
Instagram video carousel limitations:
  • Each child container must be polled to FINISHED before the parent can be created. This can take 1–5 minutes per clip.
  • Maximum 10 video slides per carousel.
  • The Instagram API may return a Fatal error on publish even when the post actually succeeded — this is handled by the verify step which checks recent media.

Clip tile anatomy

Each active slot in #cinemagraph-grid is rendered as a div.carousel-col#cine-col-{idx} (where idx is the 0-based slot index). A filled slot contains:

  • A <video> element with autoplay loop muted playsinline, src set to the clip URL, and a data-slide-idx attribute.
  • A small info button (class="cine-info-btn") that calls openCinemagraphEditModal(idx, url, prompt).

An empty slot renders as <div class="carousel-placeholder">{N}</div> (1-based slot number). During generation a spinner overlay (class="cine-spinner-overlay") is added on top of the placeholder.

Archived clip tile anatomy

Each archived entry in #cinemagraph-removed-grid is a div.carousel-col#cine-archived-{archivedIdx} with a <video> element and an info button that opens the shared #archived-info-modal via openArchivedInfoModal('cinemagraph', ...). The shared modal shows clip metadata and a "Restore to slot" dropdown.

Polling Pattern

Unlike the Carousel Generator (which uses SSE streaming), the Cinemagraph Generator uses a polling loop. The frontend calls /video-data every 20 seconds and renders any newly completed clips. 

function _cinemagraphPoll() {
    setTimeout(function () {
        fetch('/admin/articles/' + ARTICLE_ID + '/video-data')
            .then(r => r.json())
            .then(d => {
                _cinePrompts   = d.cinemagraph_prompts    || _cinePrompts;
                _cineCreatedAt = d.cinemagraph_created_at || _cineCreatedAt;

                if (d.cinemagraph_status && d.cinemagraph_status.startsWith('running:')) {
                    // Full generation still in progress
                    _cinemagraphApplyStatus(d.cinemagraph_status, null);
                    _cinemagraphUpdateGrid(d.cinemagraph_urls, _cinePrompts);
                    _cinemagraphPoll();  // schedule next poll

                } else if (d.cinemagraph_status && d.cinemagraph_status.startsWith('slide_running:')) {
                    // Single slide re-run — keep spinner on the correct slot
                    const sm = d.cinemagraph_status.match(/^slide_running:(\d+)$/);
                    if (sm) {
                        _cinemagraphSetSlideSpinner(parseInt(sm[1]), true);
                        document.getElementById('cinemagraph-status-text').textContent =
                            'Re-running slide ' + (parseInt(sm[1]) + 1) + '\u2026';
                    }
                    _cinemagraphPoll();  // schedule next poll

                } else {
                    // Done (status key deleted by worker)
                    _cinemagraphGenerating = false;
                    document.getElementById('cinemagraph-status').classList.add('d-none');
                    document.querySelectorAll('.cine-spinner-overlay').forEach(el => el.remove());

                    // If a single-slide re-run just finished, clear the old placeholder
                    // so _cinemagraphUpdateGrid will fill it with the new clip
                    if (_cineRerunningIdx !== null) {
                        const col = document.getElementById('cine-col-' + _cineRerunningIdx);
                        if (col) col.innerHTML = '<div class="carousel-placeholder">' + (_cineRerunningIdx + 1) + '</div>';
                        _cineRerunningIdx = null;
                    }

                    // Check for error result
                    if (d.cinemagraph_result && d.cinemagraph_result.startsWith('error:')) {
                        _showCinemagraphError(d.cinemagraph_result.slice(6));
                        fetch('/admin/articles/' + ARTICLE_ID + '/cinemagraph-result',
                              { method: 'DELETE' }).catch(() => {});
                    }

                    // Render completed clips
                    if (d.cinemagraph_urls && d.cinemagraph_urls.some(u => u)) {
                        _cinemagraphShowPreviews(d.cinemagraph_urls, _cinePrompts);
                    }
                    updateCinemagraphTabCounts();
                }
            })
            .catch(() => _cinemagraphPoll());  // network error — retry
    }, 20000);  // 20 second poll interval
}
Tab visibility re-sync: A visibilitychange listener is attached to document. When the tab becomes visible again after being backgrounded while _cinemagraphGenerating is true, an immediate /video-data fetch is triggered (no 20 s wait) to re-sync the UI in case generation finished while the tab was hidden.

10. End-to-End Logic Flows

A. Full Generation (all 10 slots empty)

  1. User clicks "Generate Cinemagraph". cinemagraphGenerate() fires.
  2. Frontend POSTs {prompts: _cinePrompts} to generate-cinemagraphs.
  3. Backend: loads carousel images, resolves all 10 prompts, writes them to carousel_cinemagraph_prompts, sets cinemagraph_status_{id} = "running:0/10".
  4. Backend: launches _cinemagraph_worker in a daemon thread. Returns {started: true, n_slides: 10}.
  5. Frontend: _cinemagraphShowRunning(10) — shows status bar, builds 10 placeholder tiles, starts polling loop.
  6. Worker thread, for each of 10 slides:
    1. Resolves public image URL (GCS or uploads local to GCS temp).
    2. Calls _luma_create_task() → gen_id. Sets status to "running:{i}/10".
    3. Polls every 6 s until completed, failed, or cancelled. Updates status to "running:{i}/10:dreaming" etc. as sub-status changes.
    4. On completed: downloads MP4, saves to GCS (or local). Updates cinemagraph_urls[real_idx] and writes to DB immediately.
  7. Frontend polls /video-data every 20 s. Each poll where status starts with "running:": calls _cinemagraphUpdateGrid() — slots with new URLs get video elements inserted.
  8. Worker finishes. Clears status key. Sets result to "done".
  9. Next frontend poll finds no status key → "done" branch. Hides status bar. Calls _cinemagraphShowPreviews() then updateCinemagraphTabCounts(). Generate button is hidden (all 10 slots filled).

B. Partial Generation (some slots already filled)

  1. Steps 1–4 same as Full Generation.
  2. Backend computes empty_indices — only slots that have a carousel image but no clip. If N < 10, only N clips are generated.
  3. slot_indices is passed to the worker, mapping each of the N images to their actual 0-based slot position in the 10-slot array.
  4. Worker: real_idx = slot_indices[i]. Existing clips in filled slots are preserved in cinemagraph_urls (the merge array) and are never touched.
  5. Frontend: _cinemagraphShowRunning(N) — only adds placeholder tiles for empty slots (since hasExistingClips check prevents rebuilding the grid if any video elements already exist).
  6. On completion, updateCinemagraphTabCounts() shows Generate button only if any slots remain empty.

C. Single-Slide Re-run

  1. User clicks info button on an active clip tile. openCinemagraphEditModal(idx, url, prompt) opens modal.
  2. User optionally edits the prompt and clicks "Re-run slide".
  3. cinemagraphRerunSlide(): captures oldClipUrl, oldPrompt, oldCreatedAt before the slot changes. POSTs {slide_idx: idx, prompt: newPrompt} to regenerate-cinemagraph-slide.
  4. Backend: archives old clip into carousel_cinemagraph_archived, sets cinemagraph_status_{id} = "slide_running:{idx}", launches _cinemagraph_slide_worker. Returns {started: true, archived_idx: N}.
  5. Frontend: closes modal. Calls addCinemagraphArchived(oldEntry, archivedIdx) — adds old clip to Archived tab. Resets slot to placeholder with spinner overlay. Sets _cineRerunningIdx = idx. Starts polling.
  6. Worker runs: same Luma task + poll loop as full worker. Patches only slide_idx in DB arrays.
  7. Next frontend poll after worker finishes: _cineRerunningIdx is still set → DOM clears the placeholder. _cinemagraphUpdateGrid() inserts new clip. _cineRerunningIdx = null.

D. Archive Single Clip

  1. User opens edit modal, clicks "Archive clip".
  2. cinemagraphArchiveSlide(): confirms with user. POSTs {slide_idx: idx} to archive-cinemagraph-slide.
  3. Backend: loads carousel_cinemagraphs and carousel_cinemagraph_archived. Moves the clip at slide_idx into the archived array (with metadata). Sets cinemagraph_urls[slide_idx] = null. Saves both columns.
  4. Frontend: sets _cinePrompts[idx] = null. Resets slot to placeholder. Calls addCinemagraphArchived(entry, d.archived_idx). Hides modal. Calls updateCinemagraphTabCounts() — Generate button appears.

E. Archive All

  1. User clicks "Archive All" from kebab menu.
  2. archiveAllCinemagraphs(): confirms with user. POSTs to archive-all-cinemagraphs.
  3. Backend: collects all non-null clips from cinemagraph_urls[0..9] with metadata. Appends them to carousel_cinemagraph_archived. Sets cinemagraph_urls[0..9] = [null] * 10. Saves. Returns {ok: true, archived_start_idx: N} (starting index for the newly archived items).
  4. Frontend: iterates all active clip tiles, calls addCinemagraphArchived() for each with incrementing archivedIdx, resets slots to placeholders. Clears _cinePrompts and _cineCreatedAt. Calls showCinemagraphTab('current') and updateCinemagraphTabCounts().

F. Restore Archived Clip

  1. User opens "Archived" tab, clicks info button on an archived tile. openArchivedInfoModal('cinemagraph', archivedIdx, url, prompt, meta) opens the shared archived info modal.
  2. User selects a target slot (1–10) from the dropdown and clicks "Restore to this slot".
  3. cinemagraphRestoreClip(archivedIdx, targetSlot): POSTs {archived_idx, target_slot} to restore-cinemagraph-clip.
  4. Backend: if target slot is occupied, its current clip is displaced into the archived array. The archived clip is moved from the archived array into the active slot. Saves both DB columns. Returns displacement info if applicable.
  5. Frontend: removes archived tile. If displaced, adds the displaced clip to the archived grid. Inserts restored clip into #cine-col-{slot_idx}. Updates local arrays. Calls showCinemagraphTab('current') and updateCinemagraphTabCounts().

G. Re-run All

  1. User clicks "Re-run All" from kebab menu. Modal opens with optional global prompt textarea and cost warning (~$2.50).
  2. User clicks confirm. cinemagraphRerunAll() POSTs {global_prompt: "..."} to generate-cinemagraphs.
  3. Backend: archives all existing clips (same logic as Archive All). Recomputes empty_indices — now all slots with carousel images. Resolves prompts using the global_prompt override. Launches worker.
  4. Frontend: same as Full Generation steps 5–9.
Archive-before-rerun: For "Re-run All", the backend archives all existing clips before starting generation. This means the worker always sees all slots as empty and generates fresh clips for every slide. The archived clips remain in the Archived tab.

H. Save Default Prompt

  1. User opens "Edit Default Prompt" from kebab menu. Modal shows per-article override if set, else site global default.
  2. User edits the prompt text and clicks "Save for this Article".
  3. saveCinemagraphPrompt(): POSTs {cinemagraph_prompt: val, article_id: ARTICLE_ID} to /admin/settings/save.
  4. Backend: key is cinemagraph_prompt_{article_id}. Upserted via INSERT … ON DUPLICATE KEY UPDATE. This key takes precedence over the global cinemagraph_prompt key for this article only.
  5. Frontend: updates _cineArticlePrompt = val. This value is used as the default for all future single-slide re-runs and as the fallback for any slot whose per-slide prompt is empty.
